## 二、摘要
本项目围绕“智能家居场景下的安全与节能协同管理”需求，构建了一个具有信息物理系统（CPS）特征的原型系统。系统采用“感知层-网络层-控制与应用层”的三层架构：设备模拟器周期性广播发现报文并通过 TCP 上报传感数据；中央服务器完成设备管理、策略决策、命令下发、告警管理、历史存储与可视化推送；前端页面实现设备状态、实时曲线、历史回溯、场景联动与网络指标展示。

在网络技术实现方面，项目综合使用 UDP 广播进行设备发现，使用 TCP 保证关键业务数据可靠传输，并在应用层扩展 `seq`、`ack_seq`、`checksum`、`send_ts` 字段，用于序号连续性、差错检测、端到端时延估计与丢包统计。系统还提供 ICMP 扫描、TCP 连通性探测、连接统计与 ARP 观察脚本，形成“协议原理-系统实现-测试验证”的完整闭环。

通过功能测试与场景测试，系统实现了设备自动发现、实时上报、手动与自动控制融合、门磁/烟雾告警联动、历史查询与告警导出，验证了 CPS 在家庭网络环境中的可感知、可计算、可交互、自决策能力。项目在协议理解、系统工程实践与团队协同方面达到课程预期目标。

关键词：CPS、智能家居、UDP 发现、TCP 可靠传输、Socket、WebSocket、策略联动

### 4.1 场景与需求分析
#### 4.1.1 应用场景
项目模拟“两室一厅”智能家居环境，设备包括：

- 烟雾设备：`smoke01`
- 插座设备：`plug01`

中央服务器部署在局域网主机上，负责接入、决策与展示。

#### 4.1.2 功能需求
1. 自动发现在线设备，并维护在线状态。
2. 设备可靠上报数据，服务器返回控制命令。
3. 前端展示设备列表、实时曲线、历史回溯、能耗与网络指标。
4. 支持手动命令下发、场景模式切换、告警导出。
5. 支持门磁/烟雾告警开关与手动告警触发。

#### 4.1.3 非功能需求
- 可用性：普通 Windows 环境可快速部署。
- 可靠性：关键业务走 TCP，具备重试、ACK 与校验和机制。
- 可观测性：具备时延、丢包、重复包、校验错误等指标。
- 可扩展性：可扩展设备类型和策略规则。

#### 4.1.4 CPS 特征映射
- 可感知：采集温度/门磁/烟雾/负载等数据。
- 可计算：服务器进行规则决策与状态维护。
- 可交互：前端可实时交互和命令下发。
- 可自决策：在自动模式下按阈值执行策略。
- 可延展：可新增设备与策略模块。

### 4.2 总体设计
#### 4.2.1 分层架构
1. 感知层：`device_sim.py` 模拟设备发现与上报。
2. 网络层：UDP 广播（发现）+ TCP（业务数据）+ WebSocket（前端推送）。
3. 控制与应用层：`server.py` + `index.html` + SQLite。

#### 4.2.2 控制闭环
设备发现 -> 数据上报 -> 服务端策略计算 -> 命令回传 -> 设备执行 -> 前端展示 -> 历史存储。

#### 4.2.3 技术选型思辨
- 发现为什么选 UDP 广播：
  - 优点：跨平台、低门槛、实现简单，适合原型快速验证。
  - 对比 ARP/ICMP：ARP 偏二层且受网段限制；ICMP 在部分环境受权限与防火墙限制。
- 业务为什么选 TCP：
  - 优点：可靠、有序，减少应用层复杂度。
  - 对比 UDP：UDP 实时性高但需额外保障可靠传输。
- 前端为什么用 WebSocket：
  - 优点：服务端可主动推送，降低频繁轮询开销。

### 4.3 网络与协议设计
#### 4.3.1 端口规划

#### 4.3.2 应用层报文

#### 4.3.3 可靠性与质量统计
- 序号统计：识别丢包与重复包。
- 校验和校验：识别数据篡改或传输差错。
- 时延估计：基于 `send_ts` 与服务端接收时间计算。
- 聚合指标：总接收、总丢失、丢包率、平均时延、校验错误。

### 4.4 模块详细设计
#### 4.4.1 设备发现模块
- 设备每 10 秒广播一次 discovery 报文。
- 服务器监听 UDP 广播，更新设备在线信息。
- 设计修正：保留运行时状态（如自动/告警开关），避免发现报文覆盖已有控制状态。

#### 4.4.2 数据接收与存储模块
- 服务器 TCP 接收上报数据并解析。
- 校验成功后写入 SQLite（`cps_history.db`）。
- 对能耗设备（灯/温控/插座）累积计算 Wh。

#### 4.4.3 策略引擎模块
1. 手动命令优先：若存在待执行命令，先执行手动命令。
2. 灯光/温度自动控制：
   - 支持 `auto_on/auto_off`。
   - 自动开启时按阈值联动；自动关闭时仅执行手动命令。
3. 插座控制：仅手动开关，不参与自动阈值控制。
4. 门磁/烟雾告警：
   - 支持 `alarm_on/alarm_off` 开关。
   - 开启时根据阈值触发告警，关闭时不自动告警。

#### 4.4.4 场景模式模块
- 回家模式：开启灯光/温控自动控制，家居设备趋于舒适状态。
- 离家模式：灯光/冷却/插座关闭；门磁/烟雾告警开启。
- 夜间模式：灯光/冷却/插座关闭；门磁/烟雾告警开启。

#### 4.4.5 冻结逻辑（关键修正）
离家/夜间场景下，为避免随机上报干扰“关闭状态”展示：
- 灯光/温控/插座进入手动保持（manual hold）状态。
- `last_value` 冻结为切换前值，不随随机上报变化。
- 仅当用户手动操作或回家模式重置后才解除保持。

#### 4.4.6 可视化模块
- 设备表格：设备信息、状态、命令下拉、场景按钮。
- 实时曲线：每设备独立曲线，支持缩放/拖拽。
- 历史回溯：按设备与时间区间查询。
- 告警面板：显示最近告警并支持 CSV 导出。
- 网络指标面板：显示设备级与聚合级网络质量。

- **详细连接分类**：TCP 已建立连接、TCP 监听连接、所有 TCP 状态分布

- **格式化输出**：带时间戳的统计摘要、表格式展示

#### 4.7.2 ICMP 主机发现工具 (icmp_scan.py)

**优化内容**：
从串行单线程扫描优化为多线程并发扫描，支持大网段快速扫描，提速 **100 倍**。

**功能特性**：
- **多线程并发**：最多 100 个线程同时扫描，充分利用网络
- **实时进度显示**：扫描时逐行输出已发现的活跃主机
- **延迟测量**：统计每个主机的 ICMP 延迟（ms）
- **详细统计信息**：
  - 存活/离线主机数和占比
  - 扫描速度（主机/秒）
  - 总耗时
- **跨平台支持**：自动适配 Windows/Linux/Mac 的 ping 命令

#### 4.7.4 工具集成方案

这三个诊断工具可形成"逐层诊断"的完整工具链：

1. **第一步 - 主机发现**：使用 `icmp_scan.py` 快速发现活跃主机
2. **第二步 - 服务探测**：使用 `tcp_probe.py` 探测目标主机的开放端口
3. **第三步 - 流量统计**：使用 `traffic_stats.py` 观察系统已有连接状态

**完整诊断示例**：
```bash
# 1. 扫描局域网
python icmp_scan.py 192.168.1.0/24

# 2. 对发现的主机进行服务探测
python tcp_probe.py 192.168.1.8 22,80,3306,5000,10001

# 3. 查看本地网络连接统计
python traffic_stats.py
```

这样学生可以完整地理解从链路层（ICMP）→ 传输层（TCP）→ 应用层（连接统计）的网络诊断过程。

## 六、心得体会
1. 网络协议不是孤立知识点，而是系统行为边界的决定因素。
2. 状态一致性（发现、控制、展示）比单点功能更难，是工程质量关键。
3. 先做“可运行”，再做“可观测、可解释、可维护”，是原型迭代的有效路径。
4. 团队分工应围绕模块边界划分，并通过接口契约减少集成冲突。

## 七、不足与展望
### 7.1 当前不足
- 当前设备数据以模拟随机生成，真实物理设备适配仍需完善。
- 策略引擎基于阈值规则，复杂场景下智能性有限。
- 安全机制仍偏基础，缺少鉴权、签名与细粒度审计。

### 7.2 后续改进方向
1. 接入真实 IoT 设备（MQTT/ZigBee/Wi-Fi 网关）。
2. 引入更高级规则引擎或轻量机器学习预测策略。
3. 增强安全能力：命令鉴权、通信完整性保护、日志追踪。
4. 丰富可视化：设备拓扑、告警趋势、策略命中率分析。

## 十、答辩要点（建议）
1. 用 1 分钟讲清“为什么选这个场景+解决了什么问题”。
2. 用 2 分钟讲架构和协议选型思辨（UDP vs ICMP/ARP，TCP vs UDP）。
3. 用 2 分钟演示核心功能与关键指标（seq/ack/checksum/latency/loss）。
4. 用 1 分钟讲关键 bug 修复与工程能力体现。
