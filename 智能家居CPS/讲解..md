# 智能家居CPS 详细讲解稿（含代码位置）

> 讲解顺序按“系统结构 -> 关键流程 -> 细节模块 -> 演示与答疑”组织，可直接口述。

## 一、整体架构（先讲清系统边界）
- 系统由三部分组成：设备端模拟器、服务器端、Web 可视化页面。
- 数据链路有两条：
  1) UDP 广播用于设备发现
  2) TCP 连接用于数据上报与命令下发
- 可视化使用 HTTP + WebSocket（或轮询兜底）实时展示设备状态、曲线、告警和网络指标。
- 入口启动逻辑在 [智能家居CPS/server.py](智能家居CPS/server.py#L739-L746)。

## 二、设备发现（UDP 广播）
- 设备模拟器每 10 秒广播一次 `device_id` 和 `type`，便于服务器“发现设备”。
  发包代码在 [智能家居CPS/device_sim.py](智能家居CPS/device_sim.py#L56-L65)。
- 服务器端绑定 UDP 端口 `10000` 接收广播并更新设备表。
  收包与设备表更新在 [智能家居CPS/server.py](智能家居CPS/server.py#L535-L565)。
- 发现后会通过 WebSocket 通知前端刷新设备列表。
  推送发生在 [智能家居CPS/server.py](智能家居CPS/server.py#L552-L565)。

### 口述示例
“设备端用 UDP 广播让服务器自动发现设备，服务器接收后把设备加入内存表并推送到页面，所以不用手动配置设备。”

## 三、数据上报（TCP + 应用层序号 + 校验和）
- 设备上报走 TCP 端口 `10001`，每条上报包含：`device_id`、`type`、`value`、`seq`、`send_ts`、`checksum`。
  构造 payload 位置在 [智能家居CPS/device_sim.py](智能家居CPS/device_sim.py#L92-L100)。
- `seq` 用于统计丢包/重复包，`send_ts` 用于估算端到端时延，`checksum` 用于数据校验。
  校验和算法在 [智能家居CPS/device_sim.py](智能家居CPS/device_sim.py#L20-L23) 和 [智能家居CPS/server.py](智能家居CPS/server.py#L59-L62)。
- 服务器侧处理 TCP 数据主逻辑在 [智能家居CPS/server.py](智能家居CPS/server.py#L615-L707)：
  - 解析 JSON
  - 统计丢包/重复包（基于 `seq`）
  - 校验和检测
  - 估算时延
  - 更新设备最新值和时间戳
  - 写入历史数据库
  - 触发告警逻辑
  - 返回 `ack_seq`

### 口述示例
“每条数据都带序号和校验和，服务器一边做网络指标统计，一边保证数据可靠性，还把 ACK 和控制命令返回给设备，形成闭环。”

## 四、网络指标统计（丢包率、时延、校验错）
- 每个设备的网络统计保存在 `net_stats`：包含 `rx_packets`、`lost_packets`、`bad_checksum`、`duplicate_packets`、`latency_ms_avg`。
  初始化在 [智能家居CPS/server.py](智能家居CPS/server.py#L66-L78)。
- 丢包/重复统计逻辑在 [智能家居CPS/server.py](智能家居CPS/server.py#L633-L648)。
- 平均时延统计逻辑在 [智能家居CPS/server.py](智能家居CPS/server.py#L74-L86)。
- 汇总接口 `/api/netstats` 在 [智能家居CPS/server.py](智能家居CPS/server.py#L457-L459)，前端显示在 [智能家居CPS/index.html](智能家居CPS/index.html#L472-L483)。

### 口述示例
“我们实现了端到端时延估计、丢包率、重复包统计，这些指标直接出现在页面网络指标栏。”

## 五、命令下发与控制闭环
- 前端选命令并发送给服务器 `/api/command`。
  前端发起位置在 [智能家居CPS/index.html](智能家居CPS/index.html#L334-L342)。
  后端接收与校验在 [智能家居CPS/server.py](智能家居CPS/server.py#L307-L341)。
- 服务器对命令进行设备类型校验，防止跨设备命令。
  允许命令表在 [智能家居CPS/server.py](智能家居CPS/server.py#L24-L45)。
- 命令不会立刻“推送”到设备，而是写入 `pending_cmd`，在下一次设备上报时下发。
  `pending_cmd` 的消费逻辑在 [智能家居CPS/server.py](智能家居CPS/server.py#L570-L577)。
- 设备端收到命令后更新模拟状态并打印。
  处理命令在 [智能家居CPS/device_sim.py](智能家居CPS/device_sim.py#L130-L162)。

### 口述示例
“命令是异步闭环：用户下发命令 → 服务端写入待执行命令 → 设备下一次上报时收到并执行。”

## 十四、老师常问问题（直接答法）
- Q：命令下发为什么不是立即推送？
  A：设备端是“上报驱动”的，命令在下一次上报时作为响应返回，设计更简洁且便于统计 ACK。逻辑在 [智能家居CPS/server.py](智能家居CPS/server.py#L570-L585)。
- Q：丢包率怎么统计？
  A：用应用层序号 `seq`，若跳号则认为丢包，若倒退/重复则计重复包。见 [智能家居CPS/server.py](智能家居CPS/server.py#L633-L648)。
- Q：为什么还能展示时延？
  A：设备端上传 `send_ts`，服务器收到后与当前时间差计算时延。见 [智能家居CPS/server.py](智能家居CPS/server.py#L660-L663)。
- Q：历史数据怎么存？
  A：用 SQLite，写入在 [智能家居CPS/server.py](智能家居CPS/server.py#L268-L276)，查询接口在 [智能家居CPS/server.py](智能家居CPS/server.py#L373-L401)。
